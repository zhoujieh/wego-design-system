# MVP001｜仓库管理 Figma MCP 写入计划审核

## 1. 本次验证目标

验证「真实仓库内容 → 可执行 Figma 写入计划」这条链路的可行性。具体地：从 `zhoujieh/wego-design-system` 仓库中已实现的「仓库管理」业务场景出发，只读分析其路由、场景逻辑和样式，生成一份结构化、可被 Figma MCP 消费的写入计划，作为后续自动创建 Figma 页面、状态和交互说明的输入。

本次为 MVP 验证，不调用 Figma MCP，不创建真实 Figma 文件，只产出计划文档。

## 2. 本次需求范围

**包含：**

- 仓库管理列表页（含摘要统计、卡片列表、空状态、更多操作菜单）
- 新增仓库流程（从列表页 navbar「新建」进入）
- 编辑仓库流程（从卡片更多菜单「编辑」进入）
- 删除确认流程（从卡片更多菜单「删除」进入，含二次确认弹窗）
- 空状态新增流程（无仓库时从空状态组件「新增仓库」进入）

**不包含：**

- 其他业务场景（如快捷发布产品、商家工作台、价格权限管理等）
- 库存管理
- 订单流程
- 非本次需求页面

## 3. 用户流程总览

本次计划覆盖 5 条用户流程：

1. **仓库列表浏览流程**：店主从「我的」Tab 应用中心进入仓库管理，查看摘要统计和仓库卡片列表，了解仓库总数、启用数和默认发货仓数。

2. **新增仓库流程**：店主在列表页点击导航栏右侧「新建」按钮，全屏弹层从底部滑入展示空表单（含默认值），填写仓库基础信息、履约能力、库存与发货参数后点击保存，校验通过则关闭弹层，列表顶部出现新仓库并提示「仓库已新增」。

3. **编辑仓库流程**：店主在列表页点击某张仓库卡片的「更多」按钮展开操作菜单，选择「编辑」，全屏弹层打开并回填该仓库全部字段，修改后保存，校验通过则关闭弹层，列表原位更新并提示「仓库已更新」。

4. **删除仓库流程**：店主在列表页点击卡片「更多」按钮展开菜单，选择「删除」，弹出二次确认弹窗（含仓库名称），点击「删除」确认后弹窗关闭，列表移除该卡片并提示「仓库已删除」。

5. **空状态新增流程**：当仓库列表为空时，列表区域展示空状态组件（图标+「还没有仓库」+引导文案+「新增仓库」按钮），店主点击按钮进入新增表单，流程同新增仓库流程。

## 4. Figma 页面清单

本次计划需在 Figma 中创建 4 个页面（Frame 组）：

1. **仓库管理列表页**：主任务页面，push 进入，覆盖底部 TabBar。包含顶部导航栏（返回/标题/新建）、摘要统计区、仓库卡片列表或空状态。卡片含封面图、名称、状态徽章、服务范围与发货时效摘要、更多操作按钮。是 5 条流程的公共页面。

2. **新增仓库页**：全屏弹层，从底部滑入。空表单含默认值，五个分组：仓库状态（启用/默认开关）、基础信息（名称/编码/联系人/电话/地址/定位备注/门头图）、履约能力（类型/服务范围/同城/自提/直播开关）、库存与发货（安全库存/单日上限/时效/温层/锁定/调拨）、补充信息（备注）。

3. **编辑仓库页**：全屏弹层，结构与新增页一致，标题改为「编辑仓库」，表单回填已有仓库数据。封面图使用已有链接或按名称生成示意图。

4. **删除确认弹窗**：标准 dialog，含标题「确认删除仓库？」、内容文案（含仓库名称）、取消按钮（次要）和删除按钮（危险红色）。

## 5. 页面状态清单

本次计划覆盖 9 个关键状态，每个状态对应一个独立的 Figma Frame：

1. **列表默认态**：列表页初始状态，展示 3 条模拟仓库卡片和摘要统计。触发：从应用中心进入。边界：仓库全部停用时启用数显示 0。

2. **空状态**：列表页无数据状态，展示空状态组件和新增引导。触发：仓库列表为空。边界：删除最后一个仓库后进入此状态。

3. **操作菜单展开态**：列表页某卡片下方展开编辑/删除菜单。触发：点击卡片更多按钮。边界：再次点击可收起，点击其他卡片可切换。

4. **新增仓库初始态**：新增页空表单含默认值。触发：点击新建或空状态新增按钮。边界：取消时有修改会弹出放弃确认。

5. **编辑已有数据态**：编辑页表单回填已有数据。触发：点击菜单中的编辑。边界：取消时有修改会弹出放弃确认。

6. **表单校验失败态**：表单保持当前内容，toast 提示失败原因。触发：保存时必填项缺失。边界：四种失败场景（名称/联系人/电话/地址）。

7. **保存成功态**：编辑弹层关闭，列表回填，toast 提示成功。触发：保存校验通过。边界：设为默认仓时其他仓库默认标记自动取消。

8. **删除确认态**：弹出删除确认弹窗，等待用户选择。触发：点击菜单中的删除。边界：点击遮罩或取消不删除任何数据。

9. **删除完成态**：弹窗关闭，列表移除该卡片，toast 提示已删除。触发：确认弹窗中点击删除。边界：删除最后一个仓库后进入空状态。

## 6. 交互完整性检查

- **流程闭环**：5 条流程均有明确起点和终点，保存和删除均有列表回填和 toast 反馈。
- **状态覆盖**：9 个状态覆盖了列表页的 5 个状态、新增/编辑页的 3 个状态、删除弹窗的 1 个状态，无遗漏。
- **边界情况**：每个状态均记录了边界情况，包括空状态进入条件、默认仓互斥、取消挽留、校验失败场景、删除最后一个仓库等。
- **数据流转**：保存回填（新增 unshift / 编辑原位更新）和删除移除（filter）两条数据 handoff 已明确。
- **业务规则**：默认仓互斥、启停与默认联动、必填校验、取消挽留、数据不持久化 5 条规则已记录。

## 7. 不包含内容

本次 Figma 写入计划明确不包含以下内容：

- 其他业务场景（快捷发布产品、商家工作台、价格权限管理、系统设置、产品与笔记、库存管理、交易设置）
- 库存管理与库存绑定
- 订单流程与履约单据
- 真实仓库后端服务对接
- 复杂跨仓调拨
- 地址地图选择
- 数据持久化（刷新后恢复初始模拟数据）
- 非 MVP 需求范围内的任何页面

## 8. 写入 Figma 前确认项

在将本计划作为 Figma MCP 写入输入前，请确认以下事项：

1. **Frame 命名一致性**：9 个 Frame 名称已按 `mvp001-流程编号-状态编号-页面名-状态名` 格式生成，写入 Figma 时需保持一致以便后续增量更新。

2. **状态与页面映射**：列表页承载 5 个状态（默认/空/菜单/保存成功/删除完成），新增页承载 3 个状态（初始/校验失败/保存成功），编辑页承载 3 个状态（数据/校验失败/保存成功），删除弹窗承载 1 个状态（确认）。注意保存成功态和校验失败态在新增页和编辑页均会出现。

3. **流程闭环验证**：F02 新增、F03 编辑、F04 删除、F05 空状态新增四条流程的终点均回到列表页并带有 toast 反馈，写入 Figma 时需确保状态切换路径完整。

4. **交互说明完整性**：每个 Frame 的 annotation 已包含状态名称、触发条件、页面变化、数据变化、下一步动作、边界情况六项，Figma 写入时需作为 Frame 描述或注释完整带入。

5. **后续更新同步**：本计划支持增量更新——通过 figma_frame_name 可定位到具体 Frame，新增 flow 或 state 时只需追加对应 Frame，不影响已有内容。
