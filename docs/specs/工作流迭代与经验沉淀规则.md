# 工作流迭代与经验沉淀规则

本文档由系统规则自动生成，用于人工检查。请勿直接修改；如需调整规则，应修改对应权威来源后重新生成。

## 什么时候使用

只有用户明确要求沉淀经验、补充规则、复盘形成经验或优化工作流时使用。

## 应该怎么做

- 普通反馈、自查和验收失败只用于修复当前任务，不会自动沉淀经验。
- 一次审查最多记录一条最重要、最可复用的经验。
- 记录前必须明确问题最早产生的位置、规则归属、未来落点、消费方和验收方式。
- 归属不明确时不得进入候选池。
- 同类经验只累计次数和场景证据，不重复创建。
- 快速迭代阶段当前阈值为 1 次；标准稳定阶段阈值保留为 3 次，后续只调整阈值配置即可恢复。
- 同类经验达到当前阈值时只进入等待确认状态，并询问用户是否升级。快速迭代阶段当前阈值为 1，标准稳定阶段保留 3。
- 用户未确认前不得修改正式规则，也不得登记正式场景类型。
- 正式升级前必须写清适用、不适用、例外和回退条件，并优先并入已有规则。
- 正式场景注册表只保存成熟、经过验证且会被实际消费的规则类型。

## 不能怎么做

- 不能一次批量记录多条经验。
- 不能因为业务名称不同重复创建同类候选。
- 不能在达到当前阈值时自动升级正式规则。
- 不能把候选内容写进正式场景注册表。
- 不能为技能创建并列运行时入口。

## 完成后如何检查

候选次数、场景证据、主要归属、未来规则落点、实际消费方和验收方式都完整；正式升级还必须有用户确认、场景边界和回归验证。
