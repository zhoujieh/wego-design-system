# 工作流经验沉淀与规则文档自动化计划

> 历史归档文档（不参与当前技能、规则或运行时判断）：本文所述自动生成 `specs` 规则文档的机制已删除，不能作为后续实现依据。当前唯一权威来源是 `AGENTS.md`、`.codex/skills/README.md` 与各现行 `SKILL.md`。

## 1. 目标

本次改造解决两个问题：

1. 建立一套稳定的经验沉淀机制，避免用户反馈或 AI 自查结果被直接写成正式规则，也避免经验池持续膨胀。
2. 将整个设计工作流中已经生效的规则自动提取为通俗易懂的 `specs/*.md` 文档，方便人工检查规则是否完整、清晰、重复或冲突。

本次不处理规则退出机制，也不改造现有 `specs` 文档内容。先建立经验记录、阈值触发、准确归属、正式沉淀、规则文档生成和一致性校验的完整闭环。

---

## 2. 核心原则

### 2.1 反馈不等于正式规则

用户反馈、AI 自查、验收失败或实现偏差，只能先作为经验候选记录，不能直接写入正式规则。

### 2.2 每次只记录一条最重要的经验

只有当用户明确要求“审查并沉淀经验”“补充规则”“复盘并形成经验”或“优化工作流”时，才进入经验沉淀流程。

一次审查即使发现多个问题，也只允许选择其中最重要、最有复用价值的一条进入经验池。其他问题本轮不记录、不累计，也不另建问题日志。

组件契约、UI Kit、Preview 和样式的正常迭代不受此限制。

### 2.3 归属不准确不得进入经验池

经验写入经验池前，必须先判断问题最初产生在哪个决策环节、哪个技能本应做出正确决策、未来应由哪个权威文件承载、由谁消费和验收。

不能根据最后修改了哪个文件，也不能仅根据表面组件名称判断归属。

### 2.4 出现三次才允许升级

同类经验累计出现 3 次后，才达到正式沉淀阈值。

达到第 3 次时，系统必须自动提示用户确认：

> 该经验已累计出现 3 次，达到正式沉淀阈值，是否现在将其升级为正式规则？

未得到用户确认前，不得写入正式规则。

### 2.5 正式沉淀前必须拆分场景

用户确认后，必须先明确：

- 哪些场景适用
- 哪些场景不适用
- 有哪些例外
- 条件不满足时如何回退
- 是否可并入已有规则
- 是否确实需要新增场景类型

禁止把局部经验直接扩大成“所有页面都必须怎样”的一刀切规则。

### 2.6 `specs/*.md` 只用于人工检查

`specs/*.md` 不是 AI 的运行时规则来源，也不允许作为权威规则手工维护。

它们由脚本从权威来源自动生成，用于人工查看整个工作流当前生效的规则，并检查是否完整、清晰、重复或冲突。

修改权威来源后不要求实时生成，但提交前必须保证一致。

## 3. 经验池设计

### 3.1 目录位置

在迭代技能内部新增：

```text
.codex/skills/wego-uxsystem-iterate/experience/
├── candidates.json
└── README.md
```

- `candidates.json`：AI 自动读取、匹配和更新的经验候选池。
- `README.md`：说明字段、阈值、流程和禁止事项，不存放具体经验。

普通页面生成和业务开发技能不得读取经验池。

### 3.2 候选数据结构

每条候选至少记录：

```json
{
  "id": "exp-xxx",
  "title": "抽象后的经验名称",
  "status": "observing",
  "occurrence_count": 1,
  "threshold": 3,
  "primary_owner": "wego-design",
  "secondary_owners": [],
  "ownership_reason": "",
  "problem_pattern": "",
  "scene_evidence": [],
  "applies_when_candidates": [],
  "avoid_when_candidates": [],
  "expected_rule_target": {
    "file": "",
    "field": ""
  },
  "runtime_reachability": {
    "consumer_skill": "",
    "output_field": "",
    "downstream_consumer": "",
    "acceptance_check": ""
  }
}
```

### 3.3 自动匹配与计数

每次选出本轮最重要的一条经验后：

1. 读取 `candidates.json`。
2. 判断是否与已有候选属于同一类问题。
3. 同类则不新增，`occurrence_count + 1`，并追加本次场景证据。
4. 不同类才新增候选，初始次数为 1。
5. 禁止因为业务名称不同，就重复创建实质相同的候选。
6. 达到第 3 次时，自动触发用户确认。

## 4. 归属判定标准

归属判断顺序固定为：

```text
问题在哪一步首次产生
→ 哪个技能本应做出正确决策
→ 正式规则应由哪个权威文件承载
→ 谁会在运行时消费
→ 谁负责最终验收
```

### 4.1 `wego-product`

负责：

- 用户目标理解
- 页面范围和流程拆分
- 页面、状态和信息块定义
- 业务规则与异常流程识别

典型问题：需求理解错误、流程遗漏、状态不完整、`page_spec` 内容缺失。

### 4.2 `wego-design`

负责：

- 页面范式判断
- UI Kit 匹配
- 页面布局方式
- 信息组织
- 组件选择与组合
- 设计系统消费

典型问题：页面范式错误、布局方向错误、组合关系错误、已有规则未命中、`design_consumption_plan` 决策错误。

### 4.3 `wego-ux`

负责：

- 按已确认规格实现页面
- 场景路由和宿主接入
- DOM、CSS、交互和状态转换
- 严格执行 `design_consumption_plan`

典型问题：规格正确但实现偏离、焦点切换错误、滚动或宽度实现错误、自创类名或未按契约实现。

### 4.4 `wego-tests`

负责：

- 检查规格、设计计划和实现是否一致
- 验证关键交互和状态
- 发现回归
- 正确归因问题

典型问题：验收范围缺失、关键状态未覆盖、问题归因错误、测试通过但页面明显不符合规则。

### 4.5 具体权威文件

- `components/{slug}.json`：单个组件内部结构、状态、变体和行为。
- `uikit-plan.json`：页面范式、兜底页面结构、多组件组合规则。
- `library-consumption.json`：设计系统读取、消费、复制和全局限制。
- 对应 `SKILL.md`：技能触发、职责边界、读取顺序、输入输出和交接门禁。
- `AGENTS.md`：所有技能共同遵守的仓库级硬约束。

归属无法明确时，本轮经验不得进入经验池。

## 5. 正式沉淀流程

当候选累计达到 3 次并获得用户确认后，按以下步骤执行：

1. 汇总三次出现的证据和场景差异。
2. 拆分适用场景、不适用场景、例外和回退条件。
3. 检查已有规则，优先合并、补充或增加例外，不优先新增平级规则。
4. 判断是否需要新增正式场景类型。
5. 将规则写入正确的结构化权威来源。
6. 验证运行时链路：谁读取、何时命中、影响什么决策、如何传递、如何验收。
7. 增加对应回归测试。
8. 更新设计系统版本（如涉及设计系统本体）。
9. 运行规则文档生成和一致性检查。
10. 人工检查生成后的 `specs/*.md` 是否准确表达规则。

`scenarioTypeRegistry` 只允许登记已经成熟、经过验证、会被运行时消费的正式场景类型，不得保存经验候选。

## 6. 工作流规则文档自动生成

### 6.1 输出位置

继续使用当前目录：

```text
.codex/skills/wego-design/specs/
```

不新增独立文档目录。

### 6.2 文档定位

生成文档用于：

- 查看整个设计工作流当前生效的规则
- 检查规则是否清楚、完整、重复或冲突
- 验证工作流迭代后规则是否被正确表达

生成文档不得作为页面生成 AI 的运行时规则来源。

### 6.3 文档分类

脚本应按工作职责分类生成文档，例如：

```text
specs/
├── 工作流总览与优先级.md
├── 需求理解规则.md
├── 页面设计规则.md
├── 组件与UI-Kit使用规则.md
├── 原型实现规则.md
├── 验收与回归规则.md
└── 工作流迭代与经验沉淀规则.md
```

具体文件名可根据当前已有 `specs` 内容调整，但不得新建独立目录。

### 6.4 文档写法

正文必须使用通俗易懂的中文，直接指导工作，不堆文件名、字段名和技术术语。

每条规则重点写清：

1. 什么情况下使用
2. 应该怎么做
3. 不能怎么做
4. 完成后如何检查

推荐写法示例：

```markdown
## 页面宽度不能被内容多少影响

承担主要信息展示或主要操作的区域，应默认占满可用宽度。

不能因为文本为空、字段未填写或内容较少，就让整个区域缩成一小块。页面主结构必须稳定，内容多少只能影响内部高度，不能破坏整体宽度和对齐。

以下内容可以按自身宽度展示：标签、状态文字、小型筛选项、辅助按钮、横向滚动的小卡片。

完成后检查：删除部分内容或保留空内容时，页面主体宽度是否仍然稳定。
```

文档开头只保留一段简短说明：

```text
本文档由系统规则自动生成，用于人工检查。请勿直接修改；如需调整规则，应修改对应权威来源后重新生成。
```

## 7. 自动化脚本

### 7.1 脚本位置

在仓库根目录新增一个脚本：

```text
scripts/specs.mjs
```

不拆分多个生成和校验脚本，不生成临时文件。

### 7.2 命令

```bash
node scripts/specs.mjs generate
node scripts/specs.mjs check
```

### 7.3 `generate`

- 读取所有权威规则来源。
- 按职责分类和统一文档模板生成 `specs/*.md`。
- 覆盖对应自动生成文档。
- 不修改权威来源。

### 7.4 `check`

- 使用与 `generate` 完全相同的生成逻辑，在内存中计算预期结果。
- 与仓库现有 `specs/*.md` 对比。
- 不写临时文件，不修改文档。
- 内容缺失、过期或不一致时直接失败。

### 7.5 接入现有校验

在：

```text
scripts/validate-wego-design.mjs
```

中统一调用：

```bash
node scripts/specs.mjs check
```

这样可以不要求实时同步，但提交前必须保持权威来源与 `specs` 一致。

## 9. 实施顺序

### 阶段一：建立经验池

1. 新增 `experience/` 目录。
2. 定义 `candidates.json` 结构。
3. 写明每轮只允许一条经验进入候选池。
4. 加入同类匹配、次数累计和第三次确认机制。

### 阶段二：明确归属和正式沉淀

1. 更新 `workflow-iteration.md` 的归属判断标准。
2. 明确归属不准确不得入池。
3. 增加三次阈值后的用户确认门禁。
4. 增加场景拆分和运行时可达性验证。
5. 更新同步矩阵。

### 阶段三：实现规则文档生成

1. 盘点整个工作流的权威规则来源。
2. 按职责分类规则。
3. 定义简洁、通俗、可执行的文档生成模板。
4. 实现 `scripts/specs.mjs generate`。
5. 生成首版工作流规则文档。

### 阶段四：加入一致性校验

1. 实现 `scripts/specs.mjs check`。
2. 接入 `validate-wego-design.mjs`。
3. 校验规则源变化但文档未更新时能够失败。
4. 校验手工修改生成文档时能够失败。

### 阶段五：补充测试

覆盖以下场景：

- 一次审查发现多个问题时，只取一条进入经验池。
- 同一经验再次出现时累计次数，不重复新增。
- 第三次出现时自动提示用户确认。
- 用户未确认时不得升级正式规则。
- 归属不明确时不得进入经验池。
- 正式规则必须有明确的消费和验收链路。
- 权威来源变化后 `check` 能发现 `specs` 过期。
- `generate` 后 `check` 能通过。
- `check` 不产生临时文件，也不修改仓库内容。

## 10. 完成标准

全部满足以下条件才算完成：

- 经验只能在用户明确要求审查沉淀时进入候选流程。
- 每轮最多一条经验进入候选池。
- 同类经验自动累计，达到三次自动询问用户。
- 未经用户确认不得升级为正式规则。
- 经验归属明确，能指出权威落点、运行时消费方和验收方式。
- 正式沉淀前完成适用场景、例外和回退条件拆分。
- `scenarioTypeRegistry` 不保存未成熟经验。
- 整个设计工作流规则能自动生成到当前 `specs/` 目录。
- 生成文档使用通俗中文，可直接指导工作和人工检查。
- 只维护一个 `scripts/specs.mjs`，同时支持生成和检查。
- 提交前校验能保证权威来源与 `specs` 一致。
