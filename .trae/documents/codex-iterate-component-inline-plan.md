## 拟议变更

### 一、AGENTS.md 新增 6 个小节

#### 新增 A：`任务分流`（插入在"每次开发前的读取顺序"之后）

每次收到组件相关任务，先归入以下类别：
- 已注册组件迭代
- 新增组件发布
- Token 变更
- UI Kit 同步
- 图标资产变更

如果用户只说"改一下按钮"，先判断是结构、样式、Token、文案还是展示层调整，不要直接进入编辑。

#### 新增 B：`已注册组件迭代标准步骤`（插入在"组件发布规则"之后）

10 步标准流程：
1. 按读取顺序读基础文件和组件文件
2. 确认 slug 已注册在 `components/index.json`
3. 读取 `components/{slug}.json` 和 `preview/component-{slug}.html`
4. 判断变化落点：契约结构 / 样式与状态 / Token 消费 / 预览文案 / UI Kit 复用 / inline style 语义 / 视觉核对需求
5. 修改 `components/{slug}.json`
6. 修改 `preview/component-{slug}.html`（只改标记区 `@component-css-start` 和 `@component-css-end` 之间）
7. 若样式有变化，执行 `node wegoux/scripts/extract-components-css.mjs wegoux`（仓库根执行）
8. 同步受影响的上层文档（README、SKILL.md、library-consumption.json、uikit-plan.json）
9. 递增 `wegoux/metadata.json.version`
10. 做最小必要验证并汇报风险

补充：
- 视觉细节迭代需要真实渲染核对，不要只依赖静态阅读 CSS
- 对外公开 CSS 变量前，先写入组件契约

**同时补到"组件发布规则"第 8 条**：验证新增组件是否符合移动端、微信生态、电商/工具场景。

#### 新增 C：`组件能力收敛`（紧接 B 之后）

当发现以下情况时，优先收敛而非表面修补：
- preview 里有多处 inline style 在承担组件类型、颜色、形态或定位语义
- 契约里的公开类型比真实可维护范围更散
- 宿主挂载方式、默认定位、嵌套边界没有写清楚

处理顺序：
1. 确认最小稳定公开类型
2. 确认辅助维度（tone、placement、overflow）
3. 把 inline style 里的组件语义收回 class 或契约
4. 补齐 preview 全场景示例

#### 新增 D：`预检清单`（插入在"设计质量门禁"之前）

进入编辑前，先检查：
- preview 里是否有 inline style 承担了组件语义（契约漂移）
- 当前变体是否过散，是否应该先收敛成少数稳定公开类型
- 是否涉及默认挂载方式、宿主关系、嵌套方式等组件边界变化
- 是否会连带影响 UI Kit、README、SKILL.md 或消费契约
- 如果是视觉细节迭代，是否需要打开 preview 做真实浏览器核对

#### 新增 E：`常见拦截`（插入在"预检清单"之后）

1. **用户要求直接改 `components.css`**：拦截 → 改 preview 标记区 → 执行提取脚本
2. **用户只改 preview 展示文案**：不影响结构/状态/变体时只改展示；文案反映边界变化时同步契约
3. **preview 出现 inline style 组件语义**：demo 容器布局可保留在标记区外；组件颜色/形态/定位/滚动/状态必须收回正式 class 并同步契约
4. **用户改颜色但没提 Token**：先判断是局部微调还是公共语义层；涉及公共层则补读 `colors_and_type.css` 和 `css.json`

#### 新增 F：`输入约定与输出模板`（插入在"常见拦截"之后）

**输入约定**：收到任务后补齐 5 项输入 —— 组件 slug、变更类型、是否涉及 Token、是否为新增组件、是否需要同步 UI Kit。缺少时从仓库推断，推断不了再确认。

**输出模板**（每轮回复必须包含）：
- **改了什么**：列出本轮改动的组件、文档、脚本链路
- **验证了什么**：列出执行过的脚本、校验、回归检查
- **剩余风险**：列出暂未覆盖的宿主场景、旧调用兼容性或待后续补齐项

如果信息不足，先列缺口，再说明基于什么假设继续。

| 递增版本号 | 正式迭代（契约+样式+Token+新组件） | 仅仓库管理类变更（AGENTS.md、.gitignore、Git 配置） |

1. **不删除 `.codex/` 目录**：Codex 用户仍存在，保留原 skill 作为补充参考
2. **AGENTS.md 保持中文**：所有新增内容沿用现有风格（简洁短句、列表式）
3. **不改变现有 AGENTS.md 结构顺序**：新增小节插入逻辑位置，diff 最小化

5. **本次不递增版本号**：属于仓库管理类规则/文档变更
