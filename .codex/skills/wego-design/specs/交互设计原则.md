# 交互设计原则

## 操作布局原则

- 主要操作（保存、完成）放右侧：右上角或右下角
- 确定性/前进性按钮放右侧
- 中断/返回性按钮放左侧
- 主要操作与风险操作分开放置，避免误操作

## 交互状态定义

- 默认态保持清晰、稳定、可识别。
- hover 主要服务于 PC/网页端，移动端组件不强制依赖 hover 表达。
- active / pressed 用于表达“已触发操作”的即时反馈，应足够轻快、明确。
- disabled 用于表达不可操作状态，组件应同时处理视觉反馈和交互禁用。
- 具体状态样式由组件契约、预览页和 Token 消费关系定义；本规范不展开到单组件细节。

## 页面模式与转场

移动端页面按呈现方式分为四种 primitive：**push**、**modal**、**sheet** 和 **full-screen-modal**。具体打开方式由 `design_consumption_plan.page_presentation.type` 决定，生成阶段不得临时改判。

### 全屏页面（Fullscreen）

- **特点**：占据整个屏幕，替换当前页的全部内容区域
- **适用场景**：主流程页面（首页、商品详情、我的）
- **导航方式**：通过底部 Tab 切换，或从列表页点击进入子页面
- **返回方式**：顶部导航栏返回按钮，或底部 Tab 切换

### 底部进入的模态页（Bottom-origin Modal）

- **触发**：从当前页面底部向上滑入，覆盖在原有内容之上
- **适用场景**：设置页、筛选页、选择器、详情操作面板等辅助流程
- **层级**：`--z-overlay` (500)，覆盖当前页面内容
- **关闭**：点击取消/保存按钮关闭，或通过下拉手势关闭
- **动画**：`--ease-enter`，`--duration-normal` (250ms)，从底部向上滑入
- **典型页面**：岗位权限设置、业务配置、商品筛选、客服聊天

当任务复杂度较高、需要连续浏览和修改多项设置时，允许使用**全屏模态页**变体：

- **进入方式**：仍然从底部向上进入
- **落稳状态**：覆盖整个屏幕，盖住底部 Tab
- **退出方式**：自上而下收起回原页面
- **主操作位置**：仅放在导航栏，通常使用左侧取消、右侧保存

### 底部进入模态页的实现要点

1. **导航栏**：使用 `navbar` 组件承载返回、标题和确认操作
2. **内容区**：主体优先使用已注册组件组合，不在本规范中定义组件内部细节
3. **关闭与返回**：关闭方式清晰可预期，不与主流程操作冲突
4. **层级关系**：可根据任务复杂度选择半屏或全屏；业务设置类页面优先使用全屏模态页

## 页面打开方式 primitive

以下 primitive 定义了页面间跳转的 DOM 契约和动画规范，供 `uikit-plan.json` 的 `pagePattern.presentation` 或 fallback blueprint 引用，由 `wego-ux` 按 type 分派实现。

### push

App 内二级页面跳转，替换或覆盖当前 `.phone-screen` 内的业务 screen，不离开 `wego-app/index.html`。

- **DOM**：`wego-app/index.html` 保留唯一预览壳，目标 screen 由 `scene.js` 的 template 注册
- **跳转方式**：使用 hash route、`data-route-id` 或等价原生 DOM 逻辑在 `.phone-screen` 内切换；禁止普通 `<a href="./scenes/*.html">` 或 `location.href` 顶层离开 App
- **动画**：可用轻量左右切换或无动画，不触发浏览器顶层导航
- **层级**：`overlay_level: inline`，不涉及 z-index
- **覆盖 Tab**：默认 `covers_tab_bar: false`，需要沉浸式二级页时可由设计计划明确为 true

### modal

当前页面上的轻量弹窗，不离开当前业务场景。

- **DOM**：挂载在 `.phone-screen` 内的 dialog 层，由 App 宿主统一管理
- **适用**：确认删除、短提示、短表单、风险二次确认
- **动画**：`fade` 或无动画
- **层级**：高于当前 scene，通常不覆盖底部 Tab
- **关闭方式**：明确的取消、确认、关闭按钮；不可只依赖点击遮罩

### sheet

底部弹层，从当前页面底部滑入。

- **DOM**：挂载在 `.phone-screen` 内的 sheet 层
- **适用**：筛选、选择器、批量操作、轻量设置
- **动画**：`slide-up-enter, slide-down-exit`
- **层级**：高于当前 scene；是否覆盖 Tab 由 `covers_tab_bar` 决定
- **关闭方式**：取消、完成、返回或明确关闭按钮

### full-screen-modal

全屏模态覆盖层，从底部滑入并覆盖整个屏幕。

- **DOM 契约**：
  - `wego-app/index.html` 预置或由 App JS 创建 overlay 容器
  - overlay 样式：`.phone-screen { position: relative; }`，`.modal-overlay { position: absolute; inset: 0; z-index: var(--z-overlay, 500); background: var(--bg-page); }`
  - 业务内容必须来自 `scene.js` 的 template 或宿主 `<template>`；不得依赖 `fetch()`/`XHR` 读取本地 HTML 片段
- **动画**：
  - 进入：`transform: translateY(100%)` → `translateY(0)`，`transition` 使用 `var(--duration-normal)` `var(--ease-enter)`
  - 退出：反向动画，动画结束后清空 overlay 内容
- **层级**：`overlay_level: overlay`，z-index 高于 bottom-nav
- **覆盖 Tab**：`covers_tab_bar: true`，overlay 覆盖底部导航栏
- **关闭方式**：navbar 左侧取消/返回按钮，或右上保存按钮
## 原型数据交互

- 原型必须体现真实业务数据状态和流程闭环，例如选中态、开关态、筛选结果、保存反馈、删除确认、摘要回填。
- 默认使用内存状态即可，不强制 `localStorage` 持久化。
- 只有需求明确要求“刷新后保留”或“跨会话保留”时，才实现持久化。
- 静态原型不得为了模拟接口而引入假服务或异步 fetch 本地文件；可在 `scene.js` 中定义静态数据和状态机。
