# 工作流迭代与经验沉淀规则

本文档由系统规则自动生成，用于人工检查。请勿直接修改；如需调整规则，应修改对应权威来源后重新生成。

## 经验候选硬门禁

- 用户普通反馈、AI 自查、验收失败或实现偏差不能自动进入经验池。
- 每轮最多选择一条最重要、最可复用的经验。
- 入池前必须确认问题最早产生的环节、主要归属、次要归属、权威落点、运行时消费方和验收方式。
- 归属不明确时不得入池。
- 同类经验复用已有候选，只增加 `occurrence_count` 和场景证据，不重复新增。
- 第三次出现时将状态置为 `awaiting-confirmation`，并提示用户：

> 该经验已累计出现 3 次，达到正式沉淀阈值，是否现在将其升级为正式规则？

- 用户未明确确认前，不得修改正式规则或 `scenarioTypeRegistry`。
- 正式沉淀前必须拆分适用场景、不适用场景、例外、回退条件，并检查是否可并入已有规则。
- `scenarioTypeRegistry` 只保存成熟、经过验证且会被运行时消费的正式类型，不保存经验候选。

## 候选匹配与计数

读取 `experience/candidates.json` 后执行：

1. 使用抽象后的 `problem_pattern`、主要归属、预期落点和触发条件匹配已有候选。
2. 业务名称不同但错误模式、判断条件和未来动作相同，视为同类。
3. 同类候选只增加 `occurrence_count`，并追加一条去重后的 `scene_evidence`。
4. 不同类问题才新增候选，初始次数为 1，状态为 `observing`。
5. 新增或更新时必须维护 `updated_at`；新候选同时维护 `created_at`。
6. 候选达到 3 次时，状态改为 `awaiting-confirmation`，不得直接升级。
7. 已处于 `awaiting-confirmation` 的候选再次出现时可以继续累计证据，但不得重复创建候选或绕过用户确认。

匹配不确定时不新增候选，先在最终风险中说明归属或去重仍需确认。

## 第三次确认门禁

候选从 2 次增加到 3 次时，必须向用户显示：

> 该经验已累计出现 3 次，达到正式沉淀阈值，是否现在将其升级为正式规则？

用户明确确认前：

- 不修改正式权威来源。
- 不新增 `scenarioTypeRegistry` 类型。
- 不把候选状态改为 `promoted`。
- 不生成假定已经生效的回归规则。

用户拒绝或暂不处理时，状态保持 `awaiting-confirmation`，继续保留证据。

## 正式沉淀前的场景拆分

用户确认后，先写清：

- `applies_when`：哪些结构、语义、宿主或流程条件满足时适用。
- `avoid_when`：哪些相似场景不适用。
- `exceptions`：允许偏离的例外。
- `fallback`：条件不足、规则冲突或无法命中时如何回退。
- `merge_target`：是否应并入已有规则、补充触发条件或新增例外。
- `new_scenario_type_needed`：是否确实需要新增正式场景类型。

禁止把局部经验扩大成所有页面都必须遵守的一刀切规则。

## 运行时可达性

正式规则必须完整说明：

1. 哪个技能会读取它。
2. 在流程的哪个时点命中。
3. 它影响哪个输出字段或实现动作。
4. 该决策如何传递给下游。
5. 最终由哪个验收项或守门脚本检查。

只有“写进文档”但没有消费和验收链路的规则，不得正式沉淀。
