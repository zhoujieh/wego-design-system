# 交互设计原则

## 操作布局原则

- 主要操作（保存、完成）放右侧：右上角或右下角
- 确定性/前进性按钮放右侧
- 中断/返回性按钮放左侧
- 主要操作与风险操作分开放置，避免误操作

## 交互状态定义

- 默认态保持清晰、稳定、可识别。
- hover 主要服务于 PC/网页端，移动端组件不强制依赖 hover 表达。
- active / pressed 用于表达“已触发操作”的即时反馈，应足够轻快、明确。
- disabled 用于表达不可操作状态，组件应同时处理视觉反馈和交互禁用。
- 具体状态样式由组件契约、预览页和 Token 消费关系定义；本规范不展开到单组件细节。

## 页面模式与转场

移动端页面按呈现方式分为两种模式：**全屏页面**和**底部面板**。

### 全屏页面（Fullscreen）

- **特点**：占据整个屏幕，替换当前页的全部内容区域
- **适用场景**：主流程页面（首页、商品详情、我的）
- **导航方式**：通过底部 Tab 切换，或从列表页点击进入子页面
- **返回方式**：顶部导航栏返回按钮，或底部 Tab 切换

### 底部进入的模态页（Bottom-origin Modal）

- **触发**：从当前页面底部向上滑入，覆盖在原有内容之上
- **适用场景**：设置页、筛选页、选择器、详情操作面板等辅助流程
- **层级**：`--z-overlay` (500)，覆盖当前页面内容
- **关闭**：点击取消/保存按钮关闭，或通过下拉手势关闭
- **动画**：`--ease-enter`，`--duration-normal` (250ms)，从底部向上滑入
- **典型页面**：岗位权限设置、业务配置、商品筛选、客服聊天

当任务复杂度较高、需要连续浏览和修改多项设置时，允许使用**全屏模态页**变体：

- **进入方式**：仍然从底部向上进入
- **落稳状态**：覆盖整个屏幕，盖住底部 Tab
- **退出方式**：自上而下收起回原页面
- **主操作位置**：仅放在导航栏，通常使用左侧取消、右侧保存

### 底部进入模态页的实现要点

1. **导航栏**：使用 `navbar` 组件承载返回、标题和确认操作
2. **内容区**：主体优先使用已注册组件组合，不在本规范中定义组件内部细节
3. **关闭与返回**：关闭方式清晰可预期，不与主流程操作冲突
4. **层级关系**：可根据任务复杂度选择半屏或全屏；业务设置类页面优先使用全屏模态页
